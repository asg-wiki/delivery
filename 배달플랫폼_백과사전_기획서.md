# 배달 플랫폼 업무 백과사전 — 개발 기획서

> 이 문서는 Claude Code에 전달하여 그대로 개발을 시작할 수 있도록 작성된 기획서입니다.
> 작성일: 2026-07-07 / 버전: v1.0

---

## 1. 프로젝트 개요

배달 플랫폼(쿠팡이츠, 배달의민족, 요기요, 땡겨요) 관련 업무 정보를 모아 검색할 수 있는
**정적 웹 백과사전**을 만든다. 반려 사유, 필수 서류, 정책, 프로세스, 용어 등을 등록해두고
검색 한 번으로 확인하는 사내 참고용 도구.

- 사용자: 지사/회사 전체 (링크 공유 방식)
- 배포: GitHub Pages (정적 호스팅, 서버 없음)
- 관리: VS Code + Claude Code로 데이터 파일 수정 후 git push → 자동 반영
- **1차 범위: 쿠팡이츠만 데이터 입력. 단, UI는 4개 플랫폼 확장을 전제로 설계**

## 2. 핵심 설계 원칙 (반드시 준수)

1. **오류 최소화가 최우선.** 프레임워크·빌드 도구·외부 CDN 사용 금지.
   순수 HTML + CSS + Vanilla JS만 사용. 외부 라이브러리가 꼭 필요하면
   파일을 저장소 안에 직접 포함(로컬 번들)하여 외부 네트워크 의존 제거.
2. **데이터와 화면의 완전 분리.** 내용 수정은 `data/` 폴더의 JSON만 건드리면 되고,
   화면 코드(index.html, app.js)는 손대지 않는 구조.
   초기에 수정이 매우 잦을 예정이므로 이 원칙이 중요함.
3. **확장 대비.** 플랫폼 추가(배민/요기요/땡겨요)가 데이터 파일 1개 추가 +
   설정 1줄 변경만으로 가능해야 함. 화면 코드 수정 없이.
4. **모바일 우선 반응형.** 현장에서 휴대폰으로 조회하는 경우가 많음.
5. **데이터 무결성.** JSON 스키마 검증 스크립트를 포함해, 데이터 추가/수정 시
   형식 오류를 배포 전에 잡을 수 있게 한다.

## 4. 데이터 스키마

`data/*.json`은 아래 형태의 객체 배열이다.

```json
{
  "id": "coupang-happycall-process",
  "platform": "쿠팡이츠",
  "type": "프로세스",
  "category": "해피콜",
  "title": "쿠팡이츠 해피콜 진행 절차",
  "effective_date": null,
  "content": "마크다운 형식 본문",
  "links": [
    { "label": "이미지 촬영 동의서 (구글폼)", "url": "https://..." }
  ],
  "tags": ["해피콜", "오토해피콜", "유선해피콜", "테스트주문", "인덱싱"],
  "updated": "2026-07-07"
}
```

### 필드 정의

| 필드 | 필수 | 설명 |
|---|---|---|
| id | O | 영문 소문자-하이픈. 플랫폼 접두어 포함. 전체 데이터에서 유일해야 함 |
| platform | O | "쿠팡이츠" / "배달의민족" / "요기요" / "땡겨요" 중 하나 |
| type | O | 아래 유형 목록 중 하나 |
| category | O | 자유 텍스트 소분류 (예: "해피콜", "리뷰이관", "입점서류") |
| title | O | 검색 결과에 표시되는 제목 |
| effective_date | X | 정책 적용/변경일 (YYYY-MM-DD). 없으면 null |
| content | O | 마크다운 본문. 표·리스트·굵게·주의(⚠️) 사용 가능 |
| links | X | 외부 링크 배열. 없으면 빈 배열 |
| tags | O | 검색용 동의어 포함 태그 배열. 넉넉하게 |
| updated | O | 최종 수정일 (YYYY-MM-DD) |

### type(유형) 목록 — config.js에서 관리

- `서류` — 입점·변경 등에 필요한 서류 목록
- `정책` — 본사 정책, 기준 (예: 리뷰이관 노출 기준)
- `프로세스` — 업무 절차 (예: 해피콜 진행 순서)
- `용어·상태값` — 포털 상태값, 업계 용어 설명
- `공지` — 본사 공지사항
- `안내` — 신청 방법, 링크 안내 (예: 촬영 동의서)

유형 추가가 필요하면 config.js의 배열에 1줄 추가하면 화면에 자동 반영되어야 함.

## 5. 화면 명세 (index.html 단일 페이지)

### 5-1. 상단: 플랫폼 선택 탭
- 4개 탭: **쿠팡이츠 | 배달의민족 | 요기요 | 땡겨요**
- 뱃지가 아닌 **탭 선택 방식**. 선택한 플랫폼의 데이터만 표시.
- 데이터가 없는 플랫폼 탭은 흐리게 + "준비중" 표시. 클릭 시 "준비중입니다" 안내.
- 기본 선택: 쿠팡이츠
- 탭 상태는 URL 해시(#coupang 등)에 반영 → 특정 플랫폼 링크 공유 가능

### 5-2. 검색 영역
- 큰 검색창 1개. 입력 즉시(실시간) 결과 필터링.
- 검색 대상: title + category + tags + content 본문
- 부분 일치 검색. 공백으로 구분된 여러 단어는 AND 검색 (예: "리뷰 노출")
- 한글 초성 검색(예: "ㅎㅍㅋ" → 해피콜)은 v1에서 제외, v2 후보

### 5-3. 유형 필터
- 검색창 아래 유형 칩(chip) 버튼: 전체 / 서류 / 정책 / 프로세스 / 용어·상태값 / 공지 / 안내
- 다중 선택 불가, 단일 선택 (전체가 기본)

### 5-4. 결과 목록
- 카드 리스트. 카드에 표시: 유형 라벨(색상 구분), 제목, category, 최종 수정일
- effective_date가 있으면 "적용일: 2025-07-22" 강조 표시
- 정렬: updated 최신순
- 검색 결과 0건이면 "검색 결과가 없습니다" + 등록 요청 안내 문구

### 5-5. 상세 보기
- 카드 클릭 시 카드가 펼쳐지며(아코디언) content 마크다운 렌더링
- 마크다운 렌더링은 경량 자체 구현 (지원 범위: 제목, 굵게, 리스트, 표, 줄바꿈, 링크)
  외부 CDN 라이브러리 금지. 필요 시 marked.js를 저장소에 로컬 포함하는 것은 허용.
- links 배열이 있으면 본문 하단에 버튼 형태로 표시 (새 탭 열림)
- 하단에 작게: 최종 수정일 표시

### 5-6. 디자인 가이드
- 깔끔하고 실용적인 업무 도구 느낌. 화려한 효과 불필요.
- 플랫폼별 포인트 컬러: 쿠팡이츠(파랑 계열), 배민(민트), 요기요(빨강), 땡겨요(주황)
  → 선택된 탭과 카드 상단 라인에만 은은하게 적용
- 폰트: 시스템 기본 한글 폰트 스택 (외부 웹폰트 금지)
- 모바일(375px)~데스크톱 모두 자연스럽게

## 6. 데이터 검증 (scripts/validate.js)

Node.js로 실행하는 단순 검증 스크립트. 배포 전 `node scripts/validate.js` 실행.

검증 항목:
1. 모든 JSON 파일이 파싱 가능한가
2. 필수 필드 누락 여부
3. id 중복 여부 (전체 파일 통합 기준)
4. platform / type 값이 허용 목록에 있는가
5. 날짜 형식(YYYY-MM-DD) 검사

오류 발견 시 어떤 파일 몇 번째 항목이 문제인지 명확히 출력.

## 9. 개발 순서 (Claude Code 작업 지시)

1. 폴더 구조 생성 + config.js (플랫폼/유형 목록)
2. data/coupang.json에 7절의 초기 데이터 4건 입력, 나머지 플랫폼은 빈 배열 파일
3. index.html + style.css + app.js 구현 (5절 화면 명세대로)
4. scripts/validate.js 구현 후 실행하여 데이터 통과 확인
5. README.md 작성
6. 로컬에서 index.html 열어 검색·필터·상세보기·모바일 화면 확인
7. GitHub 저장소 생성 → push → Settings에서 Pages 활성화 (main 브랜치 root)
